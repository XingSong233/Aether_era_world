状态：采用，设计蓝图
基准：`10_世界观正文/00_总纲/以太纪元世界观总纲.md`、`航行自由跃迁与开拓秩序设计蓝图.md`、`星海地理理论层级整理.md`、`10_世界观正文/10_专题设定/星际航行和开拓秩序/星海地理与星图节点.md`、`10_世界观正文/10_专题设定/星际航行和开拓秩序/航行自由跃迁与开拓秩序.md`、`10_世界观正文/10_专题设定/星际航行和开拓秩序/星际定位技术原理.md`

本文件用于设计“以太纪元”主星图、文明疆域尺度、核心星系显示节点和结构化地图数据。它不是世界观正文专题，而是为后续地图扩展、剧情新增地点、静态图制作和星域数据维护提供工作台规则。若需要判断某条地图规则属于底层公理、中间引理还是文明发展定理，先参考 `星海地理理论层级整理.md`。

正式世界内叙述整理见：`10_世界观正文/10_专题设定/星际航行和开拓秩序/星海地理与星图节点.md`。

## 一、设计目标

| 目标 | 设计口径 |
| --- | --- |
| 可视化清晰 | 主星图只显示具名核心节点，不显示全部恒星系。 |
| 保留世界规模 | 文明实际拥有数千到数万个定居、登记或影响恒星系，地图显示节点只是代表性入口。 |
| 方便结构化维护 | 节点、航线、文明、显示层级和状态字段应能整理为 JSON、数据库或后续重新设计的地图工具数据。 |
| 支持剧情新增 | 后续事件需要的新恒星系可按 `display_tier` 和 `status` 加入，不需要重画整张星图。 |
| 避免扁平地图 | 节点分布应体现悬臂、支臂交汇、暗区边缘、异常带和贸易网络，而不是简单四象限。 |

## 二、坐标范围与投影

主星图使用光年作为坐标单位，并保留三维坐标。前期可用 `x/y` 绘制 2D 图，后续可切换为 3D 星图。

| 坐标 | 范围 | 含义 |
| --- | ---: | --- |
| `x` | `-3200` 到 `+3200` | 沿当前悬臂段的延展方向。 |
| `y` | `-2100` 到 `+2100` | 从悬臂内侧到外侧，兼顾支臂交汇和暗区边缘。 |
| `z` | 常规 `-250` 到 `+250`，异常可到 `-800` 到 `+800` | 银河盘面上下偏移。 |

建议主星图视窗为约 `6500 x 4200 x 800` 光年。银心不进入主图，只作为方向标记；可在界面边缘标注“银心方向：约 18000 光年外”。

## 四、分布生成口径

节点生成应以“悬臂段 + 支臂交汇 + 暗区裂隙 + 商贸网络”为底图。

| 分布层 | 布点建议 |
| --- | --- |
| 底层恒星密度 | 使用一条轻微弯曲的悬臂带作为高密度中心，沿 `x` 延展，在 `y` 上加入正弦或样条曲线偏移。 |
| 联邦核心 | 在悬臂内侧密集带生成多个相邻簇，坐标大致偏向 `x=-2200` 到 `+600`、`y=-600` 到 `+600`。 |
| 联邦开拓臂 | 从核心区向外缘和支臂交汇处伸出 2-4 条稀疏节点链。 |
| 永恒环带 | 使用弧形或带状分布，优先落在以太场稳定、恒星密度适中、生态维护价值高的区域。 |
| 深渊裂谷 | 沿一条斜向异常带生成稀疏节点，可从 `x=+300`、`y=-1500` 延伸到 `x=+3000`、`y=-300`，并允许更大的 `z` 偏移。 |
| 商盟网络 | 不使用连续领土生成，而是在星门交汇、跨文明边境、稳定航道交叉和中立港周围生成多个贸易簇。 |
| 中立与高危节点 | 放在文明交界、悬臂间隙、暗区边缘、旧航道断裂区和遗产隔离区附近。 |

可使用一个简单的悬臂中心线作为视觉参考：

```text
arm_y = 240 * sin((x + 700) / 850) + 120 * sin(x / 1500)
```

这不是世界内公式，只是地图布点参考。正式设定中只写“主悬臂外缘与支臂交汇处的一段文明星海”。


### 星门网络与深层以太环境字段

后续地图或生成器不应把星门视为发现后立刻可用的交通点，也不应把深层以太区视为没有恒星、行星或遗迹的空白区域。建议在节点和航线数据中追加以下字段，用于表达星门修复、网络接入和深层以太环境。

| 字段 | 推荐值 | 用途 |
| --- | --- | --- |
| `ether_environment` | `stable`、`transition`、`deep_high_pressure`、`deep_boundary`、`sealed_anomaly` | 表示该节点所处以太环境；深层以太区是高扰动环境，不等于物质空间空白。 |
| `habitable_physical` | `true`、`false` | 标记若不考虑以太环境，该星系是否具有可居住或可改造条件。 |
| `stargate_presence` | `none`、`ruin`、`damaged`、`repairable`、`operational` | 区分没有星门、星门遗迹、损伤星门、可修复星门和已运行星门。 |
| `stargate_network_status` | `none`、`unidentified`、`local_calibrated`、`handshake_pending`、`networked`、`sealed` | 表示星门是否已完成身份识别、本地校准、网络握手和公共接入。 |
| `stargate_network_domain` | `unknown`、`ancient_private`、`ancient_cross_civilization`、`modern_public`、`modern_restricted`、`isolated_fragment` | 表示星门所属网络域；古代私网、跨文明星门网络和现代修复网络不能默认互通。 |
| `stargate_tier` | `none`、`local`、`regional`、`major`、`hub` | 表示星门层级；低等级星门只适合区域直连，远距离通行通常依赖更高等级星门中继。 |
| `stargate_relay_role` | `none`、`endpoint`、`relay`、`bridge`、`hub` | 表示该星门在中继链中的角色；`bridge` 用于私网与跨文明网络之间的物理桥接或隔离接口。 |
| `ancient_anchor_chain` | `none`、`suspected`、`broken`、`recoverable`、`active` | 表示古代遗留锚链状态；深层以太区中也可以存在古代锚链。 |
| `ether_current_presence` | `none`、`seasonal`、`unstable`、`stable`、`deep_current` | 表示天然以太洋流是否存在，以及是否穿过深层高扰动区域。 |
| `heritage_review_level` | `0` 到 `5` | 对遗产、肃正协议、污染和古代权限锁的审查等级。 |
| `deep_formation_pattern` | `none`、`stellar_density`、`arm_fracture`、`gravity_shear`、`ancient_network_residue`、`deep_backflow`、`catastrophe_residue`、`purification_blockade`、`swarm_activity`、`engineered_stabilization` | 表示深层以太区的主要形成规律或历史来源。 |
| `ancient_environment_state` | `unknown`、`stable`、`transition`、`locally_stabilized`、`deep_purpose_built` | 表示古文明设施在古代运行时所处的以太环境状态。 |
| `ancient_geographic_pattern` | `unknown`、`continental_stable`、`inland_sea_network`、`fragmenting_network`、`modern_islanded` | 表示该区域在古文明时代的宏观以太地理形态；用于区分古代连续大陆式场域、内海水系式高扰动区、正在断裂的网络和现代群岛化格局。 |
| `deep_drift_scale` | `none`、`decades`、`centuries`、`millennia` | 表示深层以太区边界或强度变化的典型时间尺度。 |
| `stabilization_status` | `none`、`surveyed`、`shielded`、`anchored`、`corridor_stabilized`、`locally_cleansed` | 表示文明对局部深层以太环境的影响或改造状态。 |
| `stabilization_maintainer` | 文明、机构或 `none` | 记录谁负责清洁、屏蔽、锚链稳定、隔离穹顶或共鸣中继维护。 |

生成或维护地图时，只有 `stargate_presence: operational` 且 `stargate_network_status: networked` 的节点才能作为常规星门交通节点。`repairable`、`local_calibrated` 或 `handshake_pending` 状态可以作为剧情目标、远征任务、军事争夺和开拓工程节点，但不应直接生成稳定客运或货运通道。

星门接入还应记录网络域和层级。古代存在多个文明私有星门网络，另有由各古文明共同构建和维护的跨文明星门网络；跨文明网络与各文明私网存在物理隔离，不应只用权限字段表达。生成航线时，`ancient_private` 与 `ancient_cross_civilization` 之间默认不能直接互通，除非存在 `stargate_relay_role: bridge` 的桥接枢纽或明确设定的物理中继链。低等级 `local` 星门不应生成超远距离直连；若需要远距通行，应通过 `regional`、`major` 或 `hub` 星门中继。

同一星门网络域下的两个星门，也不应默认直接形成可通行航线。两个未接触文明如果分别修复了同一古代网络域内的两个星门，首先只能说明两端具备潜在协议兼容和路由可能。只有在网络身份、星门层级、直连边或中继链都成立时，双方才可能建立诊断级握手、坐标回响、对端身份请求或探针级窗口；这类接触可以暴露“另一端也有人在修门”，但仍不等于舰船、货物或人员可稳定通行。若缺少可用中继、桥接枢纽、承压链或风险复核，两端最多确认潜在路由，不能生成公共航线。

| 判定层 | 成立条件 | 结果 |
| --- | --- | --- |
| 同网识别 | 两端恢复网络身份，确认属于同一古代网络域或可桥接网络域。 | 可能出现握手信号、坐标回响、古代协议询问或异常通讯。 |
| 路由成立 | 两端之间存在可用直连边，或所需中等级、高等级、枢纽星门中继链仍可用。 | 可以建立诊断级握手、探针级窗口或低载荷验证通道。 |
| 通行成立 | 双端完成校准、承压、权限、调度、风险复核和目的地接纳。 | 才能形成稳定舰船、货物或人员通行。 |

深层以太区节点可以同时拥有 `habitable_physical: true`、`stargate_presence: damaged`、`ancient_anchor_chain: broken` 或 `ether_current_presence: deep_current`。这表示该区域在物质层可能有宜居星球、遗迹和航道价值，但长期治理、交通接入、共鸣通讯、AI/数字人格稳定和公共救援成本显著高于稳定区。

深层以太环境不是静态背景。若节点或航线处于 `deep_high_pressure`、`deep_boundary` 或 `sealed_anomaly` 环境，应考虑记录 `deep_drift_scale`。漂移通常是长期变化，不应让公共星图每天重画边界；但在多年、数十年或跨代工程中，它会影响航图版本、空间锚寿命、星门复检周期和殖民地承认。

深层以太区也不应由地图工具均匀随机撒点生成。生成器或结构化星图应优先按 `deep_formation_pattern` 布置异常区：悬臂断裂和强引力剪切可形成裂隙；古代星门、空间锚和锚链残网可形成节点晕区或旧航道断裂带；深层回流口可形成带状走廊；大寂灭残留、肃正协议封锁和虫群长期活动可形成封存环或禁航片区；文明工程则可能在高扰动带中切出局部稳定窗口。

涉及古文明遗迹、星门、空间锚、锚链、殖民设施、研究站或封存库时，应记录 `ancient_environment_state`。当前位于深层以太区的古代设施，默认不等于古代也处于同等深层环境；多数应标为 `stable`、`transition` 或 `locally_stabilized`，表示其在古代依靠稳定环境或古代工程运行，后来才因灾害、网络崩解、锚链断裂、深层回流和长期漂移落入现代深层区。只有明确写成古代深层观测、封存、隔离或实验设施时，才使用 `deep_purpose_built`。

若某个区域用于解释古代星门网络、锚链网络或遗产设施密度，应补充 `ancient_geographic_pattern`。古文明时代的深层以太区远没有现代广泛，整体更接近连续稳定大陆中分布内海、湖泊、河流和局部险滩；现代“大海中的群岛”是灾后群岛化结果。地图数据可以把古代核心区标为 `continental_stable`，把古代高扰动但仍可绕行或沿岸通行的区域标为 `inland_sea_network`，把大寂灭前后逐渐失稳的旧航道标为 `fragmenting_network`，现代状态再由 `ether_environment` 和 `deep_formation_pattern` 表达。

文明可以通过清洁、屏蔽、空间锚、锚链稳定、隔离穹顶、共鸣中继、长期观测站和星门复检改变局部深层以太环境。地图数据应把这种影响记录为 `stabilization_status`，而不是直接把深层区改写为普通稳定区。局部稳定可能制造新航道和殖民窗口，也可能把扰动推向邻近节点，成为后续事故或跨文明争议来源。

## 六、星海可达性层级草案

状态：设计草案，暂不写入正式专题。

本层级用于描述两个空间点之间的交通关系。星海交通不是“可达/不可达”的二值状态，不同两点之间的交通代价也不只是数量差异，而是文明能力投送方式发生质变后的层级差异。同一目的地对不同文明、不同起点、不同时间窗口和不同主体可能处于不同可达层级。

### 基本层级

| 层级 | 定义 | 质变点 | 典型表现 |
| --- | --- | --- | --- |
| 物理可达 | 文明尚未实际抵达该区域，只是通过远程观测、遗产记录、物理星图或现有科学技术水平推测该区域可以抵达。 | 从未知目标变成可规划目标。 | 有坐标、天体信息或异常记录，但没有可复现航路。 |
| 航行可达 | 文明曾经抵达该区域，并保留了可复现的交通措施、航行窗口、锚定解、星门接入记录或路线档案；照旧执行上次交通措施，大体可以再次抵达。 | 从理论目标变成可重复远征目标。 | 有远征记录、返航记录、航迹残留、临时锚标或低频任务航线。 |
| 工程可达 | 文明不只可以把数个文明个体送到该区域，还能把科研、工程、补给、维护和隔离能力的一部分输送过去，支持长期驻留。 | 从一次性抵达变成持续驻留。 | 可建立科考站、前哨、维修舱、采样站、临时空间锚或长期观测设施。 |
| 制度可达 | 该区域已经被纳入文明行政和制度日常运作，拥有航图备案、许可、救援责任、审计、检疫、科研管理、边境治理或资源登记。 | 从工程存在变成文明秩序的一部分。 | 有正式管辖、登记人口或驻站机构、常规补给、责任归属和事故复盘流程。 |
| 日常可达 | 该区域被纳入大多数文明个体的实际抵达能力中，普通个体可在合理成本、手续、风险和时间内抵达。 | 从组织能力变成个体生活能力。 | 存在客运、常规货运、求学、就业、探亲、旅游、医疗转诊或普通商务通道。 |

### 使用口径

- 可达性是点对点、主体相关、时间相关的关系，不是单个星系的固有属性。
- 同一星系可以对联邦核心区为制度可达，对商盟某护航财团为工程可达，对普通个体仍不可日常抵达。
- 可达性判断必须包含返航、救援和复现能力；单次误入或不可复现抵达不足以构成航行可达。
- 可达性可以退化。深层以太漂移、星门失准、锚链断裂、肃正协议封锁、战争、虫群活动、行政撤站或保险撤销，都可能让某区域从制度可达退回工程可达、航行可达甚至物理可达。
- 非法或灰色通行应另行标注，不能直接等同于正式可达层级。例如黑市船队能抵达某地，不代表文明对普通个体提供日常可达。

### 建议结构化字段

后续若为节点、航线或起点-终点关系建立结构化数据，可补充以下字段。可达性优先记录在“关系”或“航线”数据上，不建议只写进单个节点。

| 字段 | 推荐值 | 用途 |
| --- | --- | --- |
| `accessibility_level` | `physical`、`navigational`、`engineering`、`institutional`、`daily` | 表示某主体从某起点到某终点的当前可达层级。 |
| `accessibility_actor` | 文明、机构、舰队、企业、普通个体或 `public` | 表示该层级对谁成立。 |
| `accessibility_origin` | 节点 ID 或星区 ID | 表示可达判断的出发点。 |
| `accessibility_evidence` | `observation`、`heritage_record`、`probe_return`、`repeat_route`、`outpost`、`administrative_registry`、`civilian_service` | 记录可达层级的证据来源。 |
| `repeatability` | `theoretical`、`single_success`、`repeatable`、`scheduled`、`routine` | 表示是否可复现以及是否已排程化。 |
| `sustainment_capacity` | `none`、`crew_only`、`survey_team`、`outpost`、`station`、`settlement` | 表示文明能力投送和长期驻留能力。 |
| `institutional_status` | `none`、`mission_only`、`registered`、`administered`、`public_service` | 表示该区域是否进入制度运作。 |
| `civilian_access` | `none`、`restricted`、`permit`、`subsidized`、`routine` | 表示普通个体抵达能力。 |
| `degradation_risk` | `low`、`medium`、`high`、`drifting`、`collapsing` | 表示可达层级退化风险。 |

## 七、航线数据字段

航线不应只由距离决定。每条航线应记录路线类型、制度状态和时间范围。

```json
{
  "from": "fed_core_001",
  "to": "compact_port_014",
  "route_type": "stable_anchor_chain",
  "distance_ly": 220,
  "civilian_time_days": [2, 5],
  "cargo_time_days": [5, 12],
  "permit_level": 2,
  "insurance_level": 2,
  "inspection_level": 1,
  "hazard_level": 1,
  "requires_permit": true,
  "status": "active"
}
```

### 航线类型

| `route_type` | 用途 | 时间口径 |
| --- | --- | --- |
| `stargate_corridor` | 星门主干线、核心客运、大宗贸易。 | 物理通行快，排期和检疫决定总时间。 |
| `ether_current` | 以太洋流航线、低成本商运。 | 稳定但受潮汐窗口影响。 |
| `stable_anchor_chain` | 成熟空间锚链。 | 适合区域客运、边境补给和常规贸易。 |
| `regulated_free_jump` | 许可自由跃迁。 | 快，但能源、锚定、复检和保险成本高。 |
| `frontier_anchor_chain` | 新建或低维护锚链。 | 时间浮动大，救援缺口明显。 |
| `dark_zone_probe` | 暗区或深层边缘探查航线。 | 不适合常规货运，按任务周期计算。 |
| `black_market_jump` | 非法或伪装自由跃迁链。 | 可能很快，但无正规救援、保险和事故救济。 |

## 八、航行时间估算

基础时间表应来自正式专题，后续结构化工具或地图方案可在此基础上叠加航线状态。

| 航线类型 | 典型距离 | 客运时间 | 货运时间 |
| --- | ---: | ---: | ---: |
| 同一核心星群内 | 10-80 光年 | 12 小时-3 天 | 2-6 天 |
| 稳定锚链航道 | 80-300 光年 | 2-7 天 | 5-14 天 |
| 跨星区航线 | 300-1000 光年 | 1-3 周 | 3-6 周 |
| 核心到边境 | 1000-2500 光年 | 3-8 周 | 2-4 个月 |
| 跨文明远程航线 | 1500-5000 光年 | 1-3 个月 | 3-8 个月 |
| 暗区或深层边缘探索 | 不稳定 | 3 个月-1 年以上 | 不适合常规货运 |

结构化估算时可以把总时间拆成：

```text
总时间 = 航段通行时间 + 离港排期 + 航权许可 + 能源额度调度 + 检疫安检 + 抵达复检 + 风险延误
```

军用、医疗急救、高价急件和黑市跃迁可缩短排期，但会提高能源、维护、事故、污染或法律风险。

## 九、后续新增节点规则

- 新剧情地点优先使用 `display_tier: 3` 和 `status: draft` 登记。
- 若该地点成为长期航行、政治、贸易或军事节点，再提升为 `display_tier: 2`。
- 只有文明首府、一级星门、主空间锚、舰队总港、核心生态星系或反复影响全局的节点才提升为 `display_tier: 1`。
- 新增节点应至少说明文明归属、节点类型、坐标、显示层级、风险等级和与现有节点的主要航线。
- 不要为了补满数量一次性批量命名所有普通恒星系；普通背景星系保留在实际疆域总量中即可。
- 如果一个剧情节点改变文明疆域、航行秩序、边境救援或公开地图状态，应同步考虑是否更新正式专题、发布层地图索引和版本记录。

## 十、地图版本维护

- 第一版正式主星图建议先整理 `440-460` 个节点，不需要同时生成全部实际定居恒星系。
- 旧 `20_展示发布层/30_地图/星海拓扑模拟器/` 交互原型已经废除，不再作为地图扩展基础，也不再保留旧样例数据、旧界面结构或旧显示路线。
- 后续如重建地图工具，应先从正式专题和本蓝图重新定义地图目标、数据来源、展示口径和维护责任，不沿用旧模拟器目录。
- 当正式节点数据稳定后，可另建结构化数据文件，避免把临时样本和正式星图混在一起。
- 发布层地图应通过 `source` 字段指向正式专题和本蓝图，说明哪些节点是正式采用，哪些仍是原型样本。
