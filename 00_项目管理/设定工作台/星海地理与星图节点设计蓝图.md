# 星海地理与星图节点设计蓝图

状态：采用，设计蓝图
基准：`../../02_总纲/以太纪元世界观总纲.md`、`航行自由跃迁与开拓秩序设计蓝图.md`、`../../03_专题设定/星际航行和开拓秩序/星海地理与星图节点.md`、`../../03_专题设定/星际航行和开拓秩序/航行自由跃迁与开拓秩序.md`、`../../03_专题设定/星际航行和开拓秩序/星际定位技术原理.md`

本文件用于设计“以太纪元”主星图、文明疆域尺度、核心星系显示节点和结构化地图数据。它不是世界观正文专题，而是为后续地图扩展、剧情新增地点、静态图制作和星域数据维护提供工作台规则。

正式世界内叙述整理见：`../../03_专题设定/星际航行和开拓秩序/星海地理与星图节点.md`。

## 一、设计目标

| 目标 | 设计口径 |
| --- | --- |
| 可视化清晰 | 主星图只显示具名核心节点，不显示全部恒星系。 |
| 保留世界规模 | 文明实际拥有数千到数万个定居、登记或影响恒星系，地图显示节点只是代表性入口。 |
| 方便结构化维护 | 节点、航线、文明、显示层级和状态字段应能整理为 JSON、数据库或后续重新设计的地图工具数据。 |
| 支持剧情新增 | 后续事件需要的新恒星系可按 `display_tier` 和 `status` 加入，不需要重画整张星图。 |
| 避免扁平地图 | 节点分布应体现悬臂、支臂交汇、暗区边缘、异常带和贸易网络，而不是简单四象限。 |

## 二、坐标范围与投影

主星图使用光年作为坐标单位，并保留三维坐标。前期可用 `x/y` 绘制 2D 图，后续可切换为 3D 星图。

| 坐标 | 范围 | 含义 |
| --- | ---: | --- |
| `x` | `-3200` 到 `+3200` | 沿当前悬臂段的延展方向。 |
| `y` | `-2100` 到 `+2100` | 从悬臂内侧到外侧，兼顾支臂交汇和暗区边缘。 |
| `z` | 常规 `-250` 到 `+250`，异常可到 `-800` 到 `+800` | 银河盘面上下偏移。 |

建议主星图视窗为约 `6500 x 4200 x 800` 光年。银心不进入主图，只作为方向标记；可在界面边缘标注“银心方向：约 18000 光年外”。

## 三、初始节点数量

| 分组 | 初始建议节点 | 备注 |
| --- | ---: | --- |
| 星海联邦 | 110 | 核心首府、舰队基地、星门枢纽、开拓臂和边境救援节点。 |
| 永恒环带 | 90 | 生态核心、长期维护节点、封存门户和灾害预警站。 |
| 深渊裂谷 | 90 | 实验城邦、改造殖民地、深潜前哨、异常带入口。 |
| 商盟 | 110 | 贸易港、结算枢纽、护航节点、契约自治港和中立市场。 |
| 中立、争议、暗区、遗产 | 40-60 | 仲裁港、禁航点、失联地、遗产隔离点、虫群风险点和事件节点。 |
| 总计 | 440-460 | 适合第一版可视化；后续剧情节点逐步增加。 |

这只是主星图显示层，不是恒星系总量。实际成熟定居恒星系、登记恒星系和服务覆盖恒星系按正式专题中的规模表维护。

## 四、分布生成口径

节点生成应以“悬臂段 + 支臂交汇 + 暗区裂隙 + 商贸网络”为底图。

| 分布层 | 布点建议 |
| --- | --- |
| 底层恒星密度 | 使用一条轻微弯曲的悬臂带作为高密度中心，沿 `x` 延展，在 `y` 上加入正弦或样条曲线偏移。 |
| 联邦核心 | 在悬臂内侧密集带生成多个相邻簇，坐标大致偏向 `x=-2200` 到 `+600`、`y=-600` 到 `+600`。 |
| 联邦开拓臂 | 从核心区向外缘和支臂交汇处伸出 2-4 条稀疏节点链。 |
| 永恒环带 | 使用弧形或带状分布，优先落在以太场稳定、恒星密度适中、生态维护价值高的区域。 |
| 深渊裂谷 | 沿一条斜向异常带生成稀疏节点，可从 `x=+300`、`y=-1500` 延伸到 `x=+3000`、`y=-300`，并允许更大的 `z` 偏移。 |
| 商盟网络 | 不使用连续领土生成，而是在星门交汇、跨文明边境、稳定航道交叉和中立港周围生成多个贸易簇。 |
| 中立与高危节点 | 放在文明交界、悬臂间隙、暗区边缘、旧航道断裂区和遗产隔离区附近。 |

可使用一个简单的悬臂中心线作为视觉参考：

```text
arm_y = 240 * sin((x + 700) / 850) + 120 * sin(x / 1500)
```

这不是世界内公式，只是地图布点参考。正式设定中只写“主悬臂外缘与支臂交汇处的一段文明星海”。

## 五、节点数据字段

建议节点记录采用以下字段。字段名可保持英文，方便结构化管理；显示文本使用中文。

```json
{
  "id": "fed_core_001",
  "name": "待命名核心星系",
  "civilization": "federation",
  "type": "sector_capital",
  "x": -820,
  "y": 340,
  "z": 120,
  "display_tier": 1,
  "anchor_tier": 1,
  "population_tier": 5,
  "route_importance": 5,
  "risk_level": 1,
  "tags": ["联邦核心", "舰队辖区", "空间锚"],
  "status": "active",
  "source": "03_专题设定/星际航行和开拓秩序/星海地理与星图节点.md"
}
```

### 推荐枚举

| 字段 | 推荐值 |
| --- | --- |
| `civilization` | `federation`、`ring`、`abyss`、`compact`、`neutral`、`contested`、`heritage`、`dark_zone` |
| `type` | `capital`、`sector_capital`、`stargate_hub`、`major_anchor`、`fleet_base`、`ecology_core`、`sealed_gate`、`experiment_city`、`deep_dive_outpost`、`trade_port`、`settlement`、`arbitration_port`、`heritage_isolation`、`incident_site` |
| `display_tier` | `1` 常驻核心、`2` 区域节点、`3` 剧情节点、`4` 背景节点 |
| `anchor_tier` | `0` 无锚、`1` 主空间锚、`2` 区域锚、`3` 临时锚标 |
| `population_tier` | `0` 无常住、`1` 前哨、`2` 小殖民地、`3` 成熟殖民地、`4` 大型星区、`5` 文明核心 |
| `risk_level` | `0` 稳定、`1` 常规、`2` 边境、`3` 高危、`4` 封存、`5` 禁航 |
| `status` | `active`、`frontier`、`contested`、`sealed`、`quarantine`、`incident`、`lost`、`draft` |

## 六、航线数据字段

航线不应只由距离决定。每条航线应记录路线类型、制度状态和时间范围。

```json
{
  "from": "fed_core_001",
  "to": "compact_port_014",
  "route_type": "stable_anchor_chain",
  "distance_ly": 220,
  "civilian_time_days": [2, 5],
  "cargo_time_days": [5, 12],
  "permit_level": 2,
  "insurance_level": 2,
  "inspection_level": 1,
  "hazard_level": 1,
  "requires_permit": true,
  "status": "active"
}
```

### 航线类型

| `route_type` | 用途 | 时间口径 |
| --- | --- | --- |
| `stargate_corridor` | 星门主干线、核心客运、大宗贸易。 | 物理通行快，排期和检疫决定总时间。 |
| `ether_current` | 以太洋流航线、低成本商运。 | 稳定但受潮汐窗口影响。 |
| `stable_anchor_chain` | 成熟空间锚链。 | 适合区域客运、边境补给和常规贸易。 |
| `regulated_free_jump` | 许可自由跃迁。 | 快，但能源、锚定、复检和保险成本高。 |
| `frontier_anchor_chain` | 新建或低维护锚链。 | 时间浮动大，救援缺口明显。 |
| `dark_zone_probe` | 暗区或深层边缘探查航线。 | 不适合常规货运，按任务周期计算。 |
| `black_market_jump` | 非法或伪装自由跃迁链。 | 可能很快，但无正规救援、保险和事故救济。 |

## 七、航行时间估算

基础时间表应来自正式专题，后续结构化工具或地图方案可在此基础上叠加航线状态。

| 航线类型 | 典型距离 | 客运时间 | 货运时间 |
| --- | ---: | ---: | ---: |
| 同一核心星群内 | 10-80 光年 | 12 小时-3 天 | 2-6 天 |
| 稳定锚链航道 | 80-300 光年 | 2-7 天 | 5-14 天 |
| 跨星区航线 | 300-1000 光年 | 1-3 周 | 3-6 周 |
| 核心到边境 | 1000-2500 光年 | 3-8 周 | 2-4 个月 |
| 跨文明远程航线 | 1500-5000 光年 | 1-3 个月 | 3-8 个月 |
| 暗区或深层边缘探索 | 不稳定 | 3 个月-1 年以上 | 不适合常规货运 |

结构化估算时可以把总时间拆成：

```text
总时间 = 航段通行时间 + 离港排期 + 航权许可 + 能源额度调度 + 检疫安检 + 抵达复检 + 风险延误
```

军用、医疗急救、高价急件和黑市跃迁可缩短排期，但会提高能源、维护、事故、污染或法律风险。

## 八、后续新增节点规则

- 新剧情地点优先使用 `display_tier: 3` 和 `status: draft` 登记。
- 若该地点成为长期航行、政治、贸易或军事节点，再提升为 `display_tier: 2`。
- 只有文明首府、一级星门、主空间锚、舰队总港、核心生态星系或反复影响全局的节点才提升为 `display_tier: 1`。
- 新增节点应至少说明文明归属、节点类型、坐标、显示层级、风险等级和与现有节点的主要航线。
- 不要为了补满数量一次性批量命名所有普通恒星系；普通背景星系保留在实际疆域总量中即可。
- 如果一个剧情节点改变文明疆域、航行秩序、边境救援或公开地图状态，应同步考虑是否更新正式专题、发布层地图索引和版本记录。

## 九、地图版本维护

- 第一版正式主星图建议先整理 `440-460` 个节点，不需要同时生成全部实际定居恒星系。
- 旧 `../../07_网站发布结构/30_地图/星海拓扑模拟器/` 交互原型已经废除，不再作为地图扩展基础，也不再保留旧样例数据、旧界面结构或旧显示路线。
- 后续如重建地图工具，应先从正式专题和本蓝图重新定义地图目标、数据来源、展示口径和维护责任，不沿用旧模拟器目录。
- 当正式节点数据稳定后，可另建结构化数据文件，避免把临时样本和正式星图混在一起。
- 发布层地图应通过 `source` 字段指向正式专题和本蓝图，说明哪些节点是正式采用，哪些仍是原型样本。
